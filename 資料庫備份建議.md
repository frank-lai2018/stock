# 台股資料庫備份建議（twstock，約 7GB）

> 建立日期：2026-07-12
> 目標：防止資料庫損壞 / 重灌，能快速復原。環境：Windows + PostgreSQL（本機）。

---

## 0. 先搞懂：你有「兩份來源」

| 來源 | 內容 | 重建成本 |
|------|------|---------|
| **PostgreSQL `twstock`** | 已正規化、可直接 SQL 選股的成品（約 7GB） | — |
| **`H:\data` 原始 CSV** | 股價月檔、還原檔、FinMind 基本面、Chips 快照 | DB 可由此 + 腳本重建，但要跑數十分鐘 |

**重點觀念**：DB 是「衍生資料」，理論上砍掉可用 CSV 重灌。所以真正不能丟的是 **`H:\data`**；而 DB 備份的價值在於**「快速」復原**（免重跑整條 ETL）。因此建議**兩層都備**。

---

## 1. 建議策略（分層，由重要到次要）

### 🥇 第一層：`H:\data` CSV 原始資料（終極來源，最不能丟）
- 這是所有東西的根。只要它在，DB 隨時能重建（`load_to_db.py` 全量 + `update_*` 補齊）。
- **每週**同步一份到**另一顆實體硬碟 / 外接碟 / NAS / 雲端**（不要跟 `H:` 同一顆碟，否則碟壞兩份一起沒）。
- 工具：`robocopy`（Windows 內建，增量同步）或雲端同步（OneDrive/Google Drive/rclone）。

```bat
:: 增量鏡像 H:\data → 備份碟（只複製新增/變更，刪除來源已刪的）
robocopy H:\data E:\backup\data /MIR /R:2 /W:5 /LOG:E:\backup\robocopy_data.log
```

### 🥈 第二層：PostgreSQL 邏輯備份 `pg_dump`（快速復原主力）
- 用 `pg_dump -Fc`（custom 格式，內建壓縮）。7GB 的庫壓完通常剩 ~1–2GB。
- 可跨 PostgreSQL 版本還原、可選擇性還原單表，最通用。
- **每週一次**（或每次大量匯入後）即可；日常增量已由 CSV 那層覆蓋。

```bat
:: 備份整個 twstock（含 schema、資料、pgvector 擴充宣告）
set PGPASSWORD=你的密碼
pg_dump -U frank -h localhost -d twstock -Fc -f "E:\backup\twstock_20260712.dump"
```

### 🥉 第三層（選配）：全域物件（角色/權限）
`pg_dump` 只含單一資料庫，不含角色（如 `frank` 這個 user）。重灌整台 PostgreSQL 時需要：
```bat
pg_dumpall -U postgres -h localhost --globals-only -f "E:\backup\globals_20260712.sql"
```

## 2. 還原步驟（重灌後復原）

**前置**：新裝的 PostgreSQL 要先裝好 **pgvector**（因為 `news_embedding` 用到 `vector` 型別），否則還原到該表會失敗。

```bat
:: 1) （選配）先還原角色
psql -U postgres -f "E:\backup\globals_20260712.sql"

:: 2) 建空資料庫
psql -U postgres -c "CREATE DATABASE twstock ENCODING 'UTF8';"

:: 3) 還原資料（-Fc 檔用 pg_restore）
set PGPASSWORD=你的密碼
pg_restore -U postgres -h localhost -d twstock --no-owner "E:\backup\twstock_20260712.dump"
```
- `--no-owner`：忽略原擁有者，避免角色不一致報錯（用還原時的登入者當擁有者）。
- 若 pgvector 未裝 → 先 `CREATE EXTENSION` 會失敗；先裝好擴充再還原。

**最後手段（完全沒有 dump 時）**：從 `H:\data` 重建
```bat
psql -U postgres -c "CREATE DATABASE twstock ENCODING 'UTF8';"
psql -U postgres -d twstock -f schema.sql
psql -U postgres -d twstock -f schema_rag.sql
python load_to_db.py --dsn "postgresql://frank:pwd@localhost:5432/twstock"
:: 再跑 update_chips / update_revenue / update_fundamentals 補齊近期
```

## 3. 自動化 + 輪替

### 現成工具：`backup_db.py`（一鍵，已含輪替）
本專案已提供 **`backup_db.py`**，一支搞定 pg_dump +（選配）robocopy 鏡像 + 自動輪替：
```bash
# 只做 pg_dump（保留最近 8 份）
python backup_db.py --dsn "postgresql://frank:pwd@localhost:5432/twstock" --out-dir E:\backup

# 連 H:\data 一起鏡像、並備份角色
python backup_db.py --dsn "..." --out-dir E:\backup --data-dest E:\backup\data --globals

python backup_db.py --dsn "..." --out-dir E:\backup --dry-run    # 先看會做什麼
```
它做：① `pg_dump -Fc` → `<out-dir>\twstock_YYYYMMDD_HHMM.dump`；②（給 `--data-dest` 才做）`robocopy /MIR` 鏡像 `H:\data`；③ 只保留最近 `--keep`（預設 8）份 dump；④（`--globals`）備份角色。密碼自 `--dsn` 解析並以 `PGPASSWORD` 傳給 pg_dump（免互動）。若 pg_dump 不在 PATH，用 `--pg-bin "C:\Program Files\PostgreSQL\18\bin"`。

**設成每週自動執行（Windows 工作排程器）**：
```bat
schtasks /create /tn "twstock backup" ^
  /tr "python <專案路徑>\backup_db.py --dsn \"postgresql://frank:pwd@localhost:5432/twstock\" --out-dir E:\backup --data-dest E:\backup\data" ^
  /sc weekly /d SUN /st 22:00
```

> 想的話可把備份接進 `nightly.py`；但備份建議**獨立排程**，跟每日資料更新分開，較不會互相影響。

### 密碼處理
排程不能手動輸密碼：用 `PGPASSWORD` 環境變數，或做 `%APPDATA%\postgresql\pgpass.conf`（格式 `localhost:5432:twstock:frank:密碼`），避免密碼寫死在多處。

## 4. 存放原則（3-2-1 法則）

- **3** 份資料：正本（DB/H碟）+ 兩份備份。
- **2** 種媒體：例如本機另一顆碟 + 外接碟/NAS。
- **1** 份異地：至少一份不在同一台電腦（外接碟平時拔掉、或雲端），防勒索軟體/整機故障/火災。
- 最起碼：**dump 不要跟 `twstock` 資料檔放同一顆實體硬碟**（碟壞就全沒了）。

---

## 5. 定期驗證（很重要，常被忽略）

「沒驗證過的備份 = 沒有備份」。**每季**做一次還原演練：
```bat
:: 還原到一個測試庫，確認能開、筆數正確
pg_restore -U postgres -d twstock_restore_test --no-owner "E:\backup\最新.dump"
psql -U postgres -d twstock_restore_test -c "SELECT count(*) FROM price_daily;"
```
確認後刪掉測試庫即可。

---

## 6. 不建議 / 進階（供參考，這個規模用不到）

- **PITR（Point-in-Time Recovery，WAL 歸檔）**：可還原到任意時間點，但設定複雜、佔空間，適合「不能掉任何一筆交易」的線上服務。你的資料是每日批次、且可由 CSV 重建，**用不到**，`pg_dump` 週備 + CSV 保全已足夠。
- **實體備份（`pg_basebackup`）**：整個 cluster、綁 PostgreSQL 版本，還原較死。單一資料庫用 `pg_dump` 更靈活。

---

## 7. 快速上手 checklist

- [ ] 準備一顆**非 H 碟**的備份空間（另一顆硬碟 / 外接碟 / 雲端）。
- [ ] 手動跑一次 `pg_dump -Fc` 確認能產出 `.dump`。
- [ ] 手動 `robocopy H:\data → 備份碟` 一次。
- [ ] 設定 `PGPASSWORD` 或 `pgpass.conf`，建每週 Task Scheduler 排程。
- [ ] 設定舊 dump 輪替（保留最近 8 份）。
- [ ] **做一次還原演練**，確認 dump 真的救得回來。
- [ ] 至少一份備份放異地 / 平時拔掉的外接碟。

---

### 一句話總結
> **`H:\data` 一定要另碟備份（終極來源）；再加每週 `pg_dump -Fc` 讓你能「快速」復原而不用重跑 ETL。兩層都放到非 H 碟、其中一份異地，並定期驗證還原。**
